## 1. Team Hierarchy v1.9

**Commits:** `8aef018` feat: add team hierarchy v1.9

### Finding 1.1: Missing `teamsEnabled` check on member PATCH/DELETE routes

- **Severity:** MEDIUM
- **File:** `src/app/api/teams/[id]/members/[userId]/route.ts`, lines 6–40 (PATCH), 42–65 (DELETE)
- **Description:** Both the PATCH and DELETE handlers for team members do not check `getTeamSettings().teamsEnabled` before proceeding. Every other team API route checks this flag and returns 403 if teams are disabled. These two endpoints skip the check entirely.
- **Exploitation:** If an operator disables teams, an attacker could still modify member roles or remove members via direct API calls to `PATCH /api/teams/[id]/members/[userId]` and `DELETE /api/teams/[id]/members/[userId]`.

### Finding 1.2: Child team metadata visible to non-members via parent team GET

- **Severity:** LOW
- **File:** `src/app/api/teams/[id]/route.ts`, lines 57–64
- **Description:** The GET handler for a team includes `childTeams` with `id`, `name`, `icon`, and counts of members and projects. A member of the parent team who is NOT a member of a child team can see child team names and metadata.
- **Exploitation:** A parent team member can enumerate all child teams and basic metadata without child team membership. Low severity — likely intentional for organizational visibility.

### Finding 1.3: No rate limiting on team creation

- **Severity:** LOW
- **File:** `src/app/api/teams/route.ts`, lines 24–78
- **Description:** No rate limiting or max team count enforcement. An authenticated user could create unbounded teams.
- **Exploitation:** Scripted rapid team creation for resource exhaustion.

### Positive Findings

- **Depth limit enforced:** `validateTeamDepth` in `src/lib/validations/team.ts` correctly prevents depth > 1. `parentTeamId` is excluded from `updateTeamSchema`, preventing re-parenting via PATCH.
- **Admin check on child creation:** POST route correctly calls `isTeamAdmin(userId, parentTeamId)` and returns 403 for non-admins.
- **Delete guard works:** `deleteTeam` in `src/lib/services/team-service.ts` blocks deletion when `childTeams.length > 0` and requires admin role.
- **No IDOR on writes:** All mutation operations require admin role verification.

## 2. Share Target — PWA Web Share with Metadata Fetch

**Commits:** `d7ea63d` feat: smart share target, `8cb71d1` fix: sync share target title

### Finding 2.1: SSRF — No URL validation against internal/private addresses

- **Severity:** CRITICAL
- **File:** `src/app/api/share/metadata/route.ts`, lines 5–7, 60–64
- **Description:** The Zod schema only validates that input is a syntactically valid URL (`z.string().url()`). There is no check to prevent requests to internal/private IP ranges. `fetch(url, ...)` will request `http://127.0.0.1`, `http://169.254.169.254` (cloud metadata service), `http://10.x.x.x`, etc.
- **Exploitation:** An authenticated user POSTs `{"url": "http://169.254.169.254/latest/meta-data/iam/security-credentials/"}` to read cloud instance metadata. Can also scan internal services and exfiltrate response content via returned title/description fields.

### Finding 2.2: SSRF — No protocol restriction

- **Severity:** HIGH
- **File:** `src/app/api/share/metadata/route.ts`, lines 5–7, 60
- **Description:** `z.string().url()` accepts any valid URL scheme, not just `http` and `https`. Schemes like `file://` or `ftp://` could potentially be exploited depending on the Node.js fetch implementation.
- **Exploitation:** User could attempt `{"url": "file:///etc/passwd"}`. While Node's built-in fetch typically rejects `file://`, this is an implementation detail that should not be relied upon for security.

### Finding 2.3: SSRF — Redirect following without validation

- **Severity:** HIGH
- **File:** `src/app/api/share/metadata/route.ts`, line 63
- **Description:** `redirect: "follow"` is set, so the server automatically follows HTTP redirects. An attacker can host a public URL that 302-redirects to an internal address, bypassing any initial URL validation.
- **Exploitation:** Attacker creates `https://evil.com/redirect` → 302 → `http://169.254.169.254/latest/meta-data/`. Initial URL passes validation but the server ends up fetching internal resources.

### Finding 2.4: ReDoS potential in regex-based HTML parsing

- **Severity:** MEDIUM
- **File:** `src/app/api/share/metadata/route.ts`, lines 11–16
- **Description:** `extractMetaContent` uses regex with multiple `[^>]*?` non-greedy quantifiers against attacker-controlled HTML. The 50KB read limit partially mitigates this, but adversarial HTML can still cause significant CPU time. The 5-second `AbortSignal.timeout` only covers the network fetch, not regex processing.
- **Exploitation:** Attacker hosts a page with HTML designed to trigger catastrophic backtracking.

### Finding 2.5: No rate limiting on metadata fetch endpoint

- **Severity:** MEDIUM
- **File:** `src/app/api/share/metadata/route.ts`, line 41
- **Description:** No rate limiting. Any authenticated user can trigger unlimited outbound HTTP requests from the server.
- **Exploitation:** Rapid-fire POST requests to overwhelm server outbound connections or use the server as a proxy to DDoS third-party targets.

### Finding 2.6: Response Content-Type not validated

- **Severity:** LOW
- **File:** `src/app/api/share/metadata/route.ts`, lines 66–91
- **Description:** The server fetches and processes the response body regardless of `Content-Type`. Binary files (videos, archives) are read and regex-matched. The 50KB cap and 5-second timeout provide reasonable mitigation.

### Positive Findings

- **No XSS from fetched metadata:** React auto-escapes text content. No `dangerouslySetInnerHTML` is used. Metadata title flows into `<Input value={}>` and description/URL are rendered as text. Correctly handled.

## 3. Trial Behavioral Fields + Admin API

**Commits:** `e46f277` feat: add trial behavioral fields and admin API endpoint

### Finding 3.1: Timing-attack vulnerable token comparison

- **Severity:** MEDIUM
- **File:** `src/app/api/admin/trial-stats/route.ts`, line 18
- **Description:** Token comparison uses JavaScript strict equality (`token !== process.env.TANDEM_ADMIN_API_TOKEN`), which is susceptible to timing side-channel attacks. The codebase already uses `crypto.timingSafeEqual` in the email-inbound webhook handler (`src/app/api/webhooks/email-inbound/route.ts`, line 14).
- **Exploitation:** Attacker measures response latency across many requests with varying token prefixes to reconstruct the token byte-by-byte.

### Finding 3.2: No rate limiting on admin endpoint

- **Severity:** MEDIUM
- **File:** `src/app/api/admin/trial-stats/route.ts`, line 6
- **Description:** No rate limiting. Amplifies the timing attack and enables brute-force token guessing.
- **Exploitation:** Unlimited requests to brute-force the bearer token or exploit the timing oracle.

### Finding 3.3: PII exposure in response

- **Severity:** LOW
- **File:** `src/app/api/admin/trial-stats/route.ts`, lines 54–57
- **Description:** Response includes each trial user's `id`, `email`, and `name` with no pagination. A compromised token yields the complete roster of trial users with PII in a single request.
- **Exploitation:** Compromised token → full PII dump. User IDs could target other endpoints if combined with session hijacking.

### Finding 3.4: Auth pattern inconsistency

- **Severity:** LOW
- **File:** `src/app/api/admin/trial-stats/route.ts`, lines 6–23
- **Description:** Other admin endpoints use `requireAdmin()` (session-based, checks `isAdmin` flag). This endpoint uses a separate bearer token mechanism with no audit trail, rotation mechanism, or per-user scoping. Appropriate for M2M integration with tandem-manage, but creates a parallel auth surface.

### Positive Findings

- **Auth gate is functional:** Returns 503 when `TANDEM_ADMIN_API_TOKEN` is not configured, preventing accidental exposure. Returns 401 for missing/incorrect tokens.
